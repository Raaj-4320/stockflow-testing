# Phase 0 — Firestore Schema Audit

## Security and Ownership Model
- Ownership: per-user tenant isolation via `uid`.
- Rules (`firestore.rules`):
  - `/users/{uid}` owner-only access with immutable identity constraints.
  - `/stores/{uid}` and nested subcollections require:
    - signed-in user,
    - matching `request.auth.uid == uid`,
    - verified email (`request.auth.token.email_verified == true`) for store data.
  - deny-by-default fallback for all other docs.

## Root Documents

### `users/{uid}`
- Typical fields: `uid`, `email`, `name`, `createdAt`.
- Writers: `services/auth.register`, `services/auth.login` bootstrap path.
- Readers: auth service login/bootstrap checks.

### `stores/{uid}`
- Root state fields (non-subcollection entities):
  - `profile`
  - `categories`
  - `upfrontOrders`
  - `cashSessions`
  - `expenses`, `expenseCategories`, `expenseActivities`
  - `freightInquiries`, `freightConfirmedOrders`, `freightPurchases`
  - `purchaseReceiptPostings`, `purchaseParties`, `purchaseOrders`
  - `variantsMaster`, `colorsMaster`
  - `updatedTransactionEvents`, `deleteCompensations`
  - `migrationMarkers`
- Writers: `saveData`/`syncToCloud` (merge writes), store initialization helper.
- Readers: `syncFromCloud` root doc listener.

## Active Subcollections

1. `stores/{uid}/products/{productId}`
   - product entities including variant/color stock rows.
   - read: product listener + hydration helper.
   - write: add/update/delete product flows and receive purchase flow.

2. `stores/{uid}/customers/{customerId}`
   - customer ledger entities (`totalDue`, `storeCredit`, etc.).
   - write by customer CRUD and transaction flows.

3. `stores/{uid}/transactions/{txId}`
   - sale/return/payment transactions.
   - includes settlement and return handling metadata.

4. `stores/{uid}/deletedTransactions/{recordId}`
   - deleted transaction archive with impact snapshots.

## Audit and Operational Collections

- `stores/{uid}/auditEvents/{eventId}`
  - best-effort operational and security audit entries.
- `stores/{uid}/operationCommits/{commitId}`
  - operation-level commit traces for transaction paths.
- `stores/{uid}/customerProductStats/{customerId_productId}`
  - aggregate sold/returned quantities per customer/product.

## Read Paths in Current Runtime
- `syncFromCloud()` attaches snapshot listeners to:
  - root store doc,
  - products subcollection,
  - customers subcollection,
  - transactions subcollection,
  - deletedTransactions subcollection.
- Also performs direct `getDocs` hydration for subcollections on root snapshot load.

## Write Paths in Current Runtime
- Subcollection-first writes for products/customers/transactions/deletedTransactions.
- Root `stores/{uid}` merge writes intentionally exclude subcollection-owned entities to avoid array-overwrite blast radius.
- Guarded writes require verified auth + online + hydrated state + existing store document.

## Derived/Calculated Data
- Customer due/store-credit canonical balances computed from transaction history.
- Transaction settlement and return allocation derived by helper rules.
- Finance cashbook and KPI rollups derived from transactions/expenses/delete compensations.

## Migration Markers and Assumptions
- `migrationMarkers.customerProductStatsBackfill` with expected:
  - `status: completed`
  - `version: v1`
  - `strictModeEnabled: true`
- Strict enforcement may be toggled with env flag (`VITE_ENFORCE_CUSTOMER_PRODUCT_STATS_BACKFILL`).

## Legacy Schema Variants Referenced
Used in migration scripts (not primary runtime paths):
- top-level `products` collection
- embedded `stores/{uid}.products` array

## Image/Storage Related Paths
- Product image field currently stored in product doc (`image`) and migrated to Cloudinary URLs.
- Signed upload metadata generated by API/serverless handlers.
- Migration scripts verify remaining Firebase Storage URL references.

## Verification/Auth Requirements (Data Boundary)
- Client layer: unverified users blocked in auth flow and write guards.
- Firestore rules: verified owner required for store data read/write.
